## 1. 项目摘要

“灵山小向导·灵曦”是一套面向游客与景区管理者的 AI 数字人导览系统。游客可通过
文字、麦克风、图片、兴趣偏好和位置信息与数字人交互，获得景点讲解、连续追问、
个性化路线、拍照识景和弱定位导览；系统将大模型回答按完整语义句流式送入语音合成
和 Wav2Lip 口型驱动，让数字人尽快自然开口。

管理端提供真实交互数据概览、热门咨询、知识库维护、数字人名称/形象/音色配置、
游客感受度报告以及历史游客行为分析。系统以本地 BGE-M3 + FAISS 景区知识库约束
GLM 回答，形成“游客服务—反馈采集—运营洞察—知识更新”的持续优化闭环。

本项目针对赛题提出的四类核心痛点给出对应方案：

| 赛题痛点 | 产品应对 |
|---|---|
| 旺季导游资源稀缺 | 7×24 小时在线数字人，多终端访问，统一知识口径 |
| 传统录音导览单向 | 支持文字、语音、图片、位置和连续追问 |
| 缺乏情感连接 | 真人数字人口型、自然语音、情绪状态和轻量模式降级 |
| 管理者存在数据盲区 | 实时运营看板、热门主题、情绪趋势、满意度与服务建议 |

## 3. 用户与业务场景

### 3.1 游客角色

1. **首次到访游客**：不了解景区结构，希望获得入园后的顺路路线。
2. **亲子家庭**：关注轻松、有趣、少折返和休息点。
3. **文化兴趣游客**：希望深入了解佛教文化、建筑和历史背景。
4. **自由行游客**：不愿跟团，希望随时提问、拍照识景和连续追问。
5. **特殊环境游客**：GPS 或网络不稳定时，需要二维码或手动定位降级。

### 3.2 管理者角色

1. 运营人员查看实时服务量、热门咨询和响应情况。
2. 内容人员更新讲解资料和常见问答，无需修改程序。
3. 服务主管查看游客情感、满意度和重点投诉方面。
4. 决策人员结合 140447 条历史游客记录进行客群与景点分析。

### 3.3 核心体验旅程

```text
入园打开网页/Android App
    → 选择历史/自然/亲子兴趣
    → 文字或麦克风提问
    → 本地知识库检索与引用
    → 数字人语音、口型、表情同步讲解
    → 拍照识景 / 获取位置 / 生成路线
    → 提交评分与建议
    → 管理后台形成运营洞察
    → 更新知识库和服务策略
```

## 4. 产品功能设计

### 4.1 游客端

- **景区问答**：支持事实问题、文化故事、游览建议和会话追问。
- **麦克风交互**：浏览器通过 HTTPS 获取权限，录音完成后自动识别并进入统一问答。
- **数字人讲解**：数字人按语义句开始播报，不等待整段文字生成完成。
- **个性化路线**：基于游客兴趣、时间和当前位置组织路线及讲解重点。
- **拍照识景**：视觉模型先识别场景，再由本地知识库校准讲解，降低景点幻觉。
- **弱定位导览**：支持当前位置、景点码和手动选择；定位失败仍可继续使用。
- **满意度反馈**：游客可对具体景区/景点提交 1–5 分与文字建议。
- **Android 端**：网页封装 APK，统一复用 HTTPS 麦克风、问答和数字人服务。

### 4.2 管理端

- **数据概览**：今日游客、近七日游客、累计问答、平均响应和热门主题。
- **知识库管理**：官方资料只读，管理员资料可上传、更新、删除并触发索引重建。
- **数字人管理**：配置显示名称、已安装形象、TTS 音色和默认表情，新会话生效。
- **游客感受度**：区分七类音视频情绪、文本倾向、方面和显式满意度，展示分析模式。
- **历史游客洞察**：分析 2025 年 140447 条访问记录、50000 名游客和 152 个景点。
- **智能建议**：根据真实咨询与反馈证据生成服务优化建议，不伪造满意度数据。

## 6. 核心技术设计

### 6.1 可靠景区 RAG

1. 官方 DOCX、TXT、XLSX 和管理员文档按标题与段落切片。
2. BGE-M3 将知识片段编码为 1024 维归一化向量。
3. FAISS `IndexFlatIP` 执行余弦相似度检索。
4. 命中景点名称或别名后先做元数据过滤，减少兄弟景点信息串扰。
5. 系统提示约束数字、日期、重量等事实必须来自检索上下文。
6. 资料不足时明确回答“当前知识库未覆盖”，避免大模型编造。
7. 引用 ID、来源和小节随回答返回，便于评委和管理员追溯。

RAG embedding 采用“CPU 常驻协调器 + 按需 GPU 子进程”。平时 BGE-M3 只占 CPU
内存；检索时从 0–3 号 GPU 中选择空闲卡，GPU worker 空闲 180 秒后退出，从而彻底
释放 CUDA 上下文。连续问答复用同一 worker，兼顾首轮速度与资源利用率。

### 6.2 低延迟语音数字人

```text
浏览器录音
  → GLM-ASR
  → RAG/GLM 流式文本
  → 按句号、问号等语义边界切分
  → GLM-TTS 流式 PCM 合并成完整语义段
  → WAV 上传 LiveTalking
  → Wav2Lip FP16 推理
  → WebRTC 音视频
```

系统不按固定短字数粗暴切分，而是优先等待完整语义句；同一语义段的 PCM 块合并后
再进入数字人，避免每个提供方分块重复重采样、淡入淡出和吞字。再次提问时会取消旧
播报队列，避免新旧答案重叠。

### 6.3 多模态识景与情绪分析

- 图片由 GLM-4V 识别，结果再次进入景区 RAG 校准。
- 游客授权后，HumanOmni 输出 angry、disgust、fear、happy、neutral、sad、surprise。
- 文本模块提取讲解内容、回答准确性、响应速度、路线、票价、餐饮和交通等方面。
- 七类情绪、文本倾向和 1–5 分满意度分别存储，不把历史满意度伪装成情绪标签。
- 情绪分析异步执行，失败时明确标记文本降级，不阻塞实时问答。
- 默认在分析后删除原始媒体，仅保留结构化结果。

### 6.4 按需 GPU 与稳定性

- LiveTalking 权重在 CPU/RAM 待机，浏览器建立 WebRTC 后自动选择空闲 GPU。
- 页面关闭或心跳失效后，数字人进程退出以释放 CUDA 上下文。
- RAG embedding 检索后保留 GPU 180 秒，连续问答直接复用；随后 worker 退出。
- 本地 Qwen2-7B 仅选择本地模型路线时加载，空闲后自动卸载。
- RAG、API、LiveTalking 和本地模型均有健康检查和 watchdog。
- LiveTalking 不可用时游客端回退到 Live2D 轻量模式，文字问答仍可继续。

## 8. 测试与验证

### 8.1 关键指标

| 测试项 | 结果 | 结论 |
|---|---:|---|
| 标准事实题 | 15/15，100% | 超过赛题 90% 要求 |
| 语音结束→首个非静音数字人音频 | 3373ms | 满足小于 5 秒目标 |
| 质量优先热链路首响中位数 | 3618ms | 三轮均收到完整非静音音频 |
| LLM 首 token | 535ms | 流式生成有效 |
| 首 token→首响 | 3121ms | 语音与数字人阶段可量化 |
| API 回归 | 35 passed | 核心接口通过 |
| 语音回归 | 10 passed | PCM 合并与数字人队列通过 |
| RAG pipeline/session | 9 passed | 会话、检索与流式处理通过 |

性能数字均对应本机热链路测试。公网环境会叠加页面加载、TURN 网络 RTT、首次 GPU
迁移和云模型波动；答辩时应说明测试口径，不把局域网结果包装成所有网络环境保证值。

### 8.2 稳定性与安全验证

- 模型密钥只保存在服务端，不下发浏览器。
- RAG 与 LiveTalking 只监听内部地址，由网关代理。
- 管理后台采用独立端口、签名会话 Cookie 和来源校验。
- 知识文档修改、索引重建只允许已登录管理员操作。
- 文件类型、大小、会话长度和历史轮数均有限制。
- 多模态原始媒体默认分析后删除。

## 9. 创新点

1. **首句即说的跨模型流水线**：将串行“全文完成后播报”改造成 LLM、TTS、数字人按语义句衔接。
2. **视觉结果二次 RAG 校准**：通用视觉模型负责识别，本地资料负责最终景点事实讲解。
3. **事实级可追溯与拒答**：引用资料、景点元数据过滤、会话边界和缺失信息拒答共同降低幻觉。
4. **按需 GPU 生命周期**：数字人、RAG embedding 和本地 LLM 分别按需启用，闲时释放显存。
5. **位置多级降级**：GPS 不可用时仍可通过景点码、Wi-Fi/二维码适配层或手动选择继续导览。
6. **实时与历史双层洞察**：把当日互动和赛题 14 万条历史记录放在同一运营决策界面。
7. **证据化感受度报告**：保留模态、置信度、降级方式和原始方面，不混淆情绪与满意度。

## 11. 项目边界与后续规划

### 11.1 当前边界

- 当前使用已安装的 Wav2Lip 真人头像；新增外观需要制作新的头像模型包。
- GPS/Wi-Fi 为演示适配层，生产环境需接入真实坐标、地图和锐捷网络位置能力。
- 默认问答、语音和视觉依赖云端 GLM，公网波动会影响端到端时延。
- 自建 HTTPS 当前可使用自签名证书，正式运营需配置可信证书和域名。
- 高并发数字人需要进一步增加 GPU 配额、会话队列和容量压测。

### 11.2 演进路线

1. 接入景区 POI、室内 Wi-Fi 定位和拥挤度，动态规划路线。
2. 增加多景区租户、方言音色、儿童模式和无障碍模式。
3. 建立数字人头像制作流水线，支持景区 IP 与服装模型包。
4. 引入边缘节点缓存与模型量化，提高旺季并发并降低云服务成本。
5. 建立真实游客授权评测集，持续评估事实准确率、自然度与情绪识别。
