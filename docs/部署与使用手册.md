# 部署与使用手册

## 1. 环境

- 当前演示环境：本地 4×NVIDIA RTX 3090 服务器运行模型与业务服务，华为云服务器作为
  公网入口转发 HTTPS、管理后台及 WebRTC/TURN 流量；华为云节点不承担模型推理。
- Linux、NVIDIA GPU、CUDA 12.x
- Conda 环境：`ccc`（导览 API/LiveTalking）和 `softcup`（BGE-M3/FAISS RAG）
- Python 3.12
- 可用的智谱 API Key，存放在根目录 `softcup_glmkey` 或环境变量 `ZHIPU_API_KEY`
- 下载模型时可使用 `http://127.0.0.1:7890` 代理；运行期访问本地服务必须设置 `NO_PROXY`

该部署结构仅描述当前比赛演示环境，系统也可迁移至景区私有服务器或云端 GPU，
不要求必须使用 4 张 RTX 3090。

## 2. 启动服务

```bash
source /home/anaconda/etc/profile.d/conda.sh
conda activate ccc
cd /home/gmn/codes/cup

# 权重常驻 CPU/RAM，会话建立时自动选择 GPU 0–3 中的空闲卡。
LIVETALKING_CPU_STANDBY=true bash deploy/start_livetalking.sh
bash deploy/start_api.sh
```

端口说明：

| 端口 | 用途 | 是否直接提供给游客 |
|---|---|---|
| 8001 | HTTP 游客端、开放 API | 是，文字功能可用 |
| 8443 | HTTPS + TURN/TCP，游客页面及 WebRTC 媒体中继 | 是，麦克风和远程访问推荐 |
| 8010 | LiveTalking WebRTC/口型内部服务 | 否，由 8001/8443 代理 |
| 8020 | BGE-M3 + FAISS + GLM RAG 服务 | 否，仅监听 127.0.0.1 |
| 8444 | 独立 HTTPS 管理后台 | 是，仅供管理员 |
| Unix socket `deploy/vision-clip.sock` | SigLIP/CLIP 识景图像编码服务（softcup，可选） | 否，API 进程通过 socket 调用 |

## 3. 使用入口

- 游客端：`http://192.168.200.27:8001/`
- 麦克风端：`https://192.168.200.27:8443/`
- 管理端：`https://192.168.200.27:8444/admin`
- API 文档：`http://192.168.200.27:8001/docs`

HTTPS 首次打开需要接受自签名证书。浏览器如果运行在其他电脑，`localhost` 指向那台
电脑自身；可使用服务器 IP，或仅转发 8443。服务会在同一端口自动区分 HTTPS 与
TURN/TCP，因此远程浏览器的 LiveTalking 媒体也能通过 TCP 隧道。

## 4. 游客端操作

1. 选择历史文化、自然风光或亲子家庭兴趣。
2. 输入问题后选择“文字问答”，或在 HTTPS 页面点击“点击说话”。
3. 可上传音频、拍照识景、请求当前位置讲解和个性化路线。
4. 数字人会在首句生成后开始播报；再次提问会打断上一段播报。
5. 游览结束可选择灵山胜境或拈花湾的具体子景点，提交 1–5 分评价和文字建议。

## 5. 管理端操作

- 数据概览：查看人次、问答、延迟、情感和热门景点。
- 知识库：上传 `.docx`、`.txt`、`.md`；同名文件视为更新；官方资料只读。
- 数字人：修改名称、选择已安装形象和 TTS 声音；服装随形象素材确定，表情由对话情绪策略实时驱动。
- 感受报告：查看近期问题、评价与系统生成的服务改进建议。
- 多模态感受度：查看游客文字/语音对话自动产生的七类情绪、满意倾向、方面和模型/
  降级状态。管理员无需上传媒体；游客端授权语音分析后，原音频默认在任务结束时删除。
- 历史游客洞察：查看赛题 XLSX 的满意度、月份、景点类型、年龄、消费和灵山样本分析。

管理端使用独立端口和服务端签名的 HttpOnly/Secure 会话 Cookie。公网部署前必须
修改默认管理员密码，并配置固定的 `ADMIN_SESSION_SECRET`；知识文档修改和索引重建
不能从游客端口调用。

### 5.1 情感模型配置

优先推荐将微调模型做成独立 HTTP 推理服务，并设置 `EMOTION_INFERENCE_URL`；也可
配置 `EMOTION_MODEL_PATH`、`EMOTION_BASE_MODEL_PATH`、`EMOTION_BERT_PATH`、
`EMOTION_INFERENCE_SCRIPT`、`EMOTION_PYTHON` 和 `EMOTION_GPU` 由 API 调用本地
适配器。当前 LoRA 默认在 `/home/gmn/codes/cup/model/emotion_v5_stage2`，运行环境使用
`softcup`。仅复制 LoRA 不足以运行：还需训练脚本所用的 `emotion_stage1` 多模态基座，
且指定的 Python 必须能导入训练时修改过的 `humanomni`（含
`emotion_probs_from_logits`）、`peft` 和 `torch`。仅有官方 HumanOmni 源码不足以
解析该模型的七分类分数。启动后先
在管理后台检查真实状态；环境缺失时系统执行 ASR 文本降级分析。

历史 XLSX 默认读取 `资料/示范景区公开资料包/景点景区旅游数据行为分析数据.xlsx`，
可通过 `TOURISM_DATASET_PATH` 覆盖。`start_api.sh` 启动时按文件指纹将明细幂等导入
SQLite，并解析/缓存聚合结果；源文件更新后会重导明细及失效缓存。

## 6. 健康检查与测试

```bash
curl http://127.0.0.1:8001/health
curl http://127.0.0.1:8020/health
curl http://127.0.0.1:8001/v1/livetalking/status

cd /home/gmn/codes/cup/services/api
python scripts/eval_accuracy.py
```

预期准确率不低于 90%。当前 15 题冒烟基线为 100%；赛题提交前须补充专家复核测试集。

管理员登录后还应验证：实时概览可加载、历史记录数与 XLSX 一致、情感模型状态真实，
以及从游客端录制一段已授权语音完成 `queued → processing → completed/failed` 全链路。
模型缺失时看到 `text-fallback` 属于预期降级，不等同于多模态模型验证通过。

## 7. 日志与排障

- API：`deploy/api.log`
- HTTPS API：`deploy/api-ssl.log`
- RAG：`deploy/rag.log`
- RAG 健康看护：`deploy/rag-watchdog.log`
- 数字人：`deploy/livetalking/service.log`

常见问题：

- `403 Forbidden`：不要访问 8010 的目录；产品入口是 8001 或 8443。
- `ERR_EMPTY_RESPONSE`：对应端口没有监听，先检查 `ss -lntp | rg '8001|8443|8010|8020'`。
- RAG 首次启动失败：检查 `deploy/rag.log`、BGE-M3 本地缓存和 `RAG_GPU`；已有模型缓存时保持 `RAG_HF_OFFLINE=true`。
- `CUDA out of memory`：完整 Qwen2-7B 与轻量 Qwen3-1.7B 分别从候选卡中动态选择至少 18 GiB、6 GiB 空闲显存的 GPU。若启动前没有达标卡，页面会显示门槛和各卡情况；若加载或推理期间发生真实 OOM，页面会明确显示 CUDA OOM，并建议改用轻量路线或云端路线。详细日志分别为 `deploy/local-llm.log`、`deploy/local-llm-lite.log`。
- 麦克风不可用：浏览器要求 HTTPS 或 localhost，请使用 8443。
- 数字人无视频：确认 `/v1/livetalking/status` 中 `ready=true`，再检查浏览器 WebRTC。
