# 文字到语音端到端延迟测试与优化

测试日期：2026-07-19

## 2026-07-20 首响优化

- 最终采用“语义短句 + 连续 PCM”链路：首个不少于 8 字的完整逗号短句立即进入
  GLM-TTS，后续短句仍按 24 字阈值控制，兼顾首响与自然停顿。
- GLM-TTS 返回的 PCM 块经过原有水印音过滤后立即推送到 LiveTalking；数字人使用
  有状态重采样器连续接收，只在整次回答首尾做一次 5ms 淡入淡出，避免句子边界的杂音、
  微停顿和吞字。
- GLM API 改用持久 HTTP 连接，减少重复 TCP/TLS 建连；页面加载时后台预热 RAG，
  RAG 与 LiveTalking 均在最后活动 180 秒后释放 GPU。
- 清理了占用 8001/8443/8444 的旧孤儿进程，当前 API、RAG、LiveTalking 与 TLS
  转发全部由 `cup-api-stack.service` 同一 cgroup 托管。
- 最终连续四轮首响为 2705ms、2032ms、1909ms、2288ms，中位数 2160ms、
  最大值 2705ms，全部低于 3 秒目标；同轮后台 RAG 冷预热为 1916ms，不阻塞页面操作。
- 云端主模型由首 token 波动明显的 `glm-4.5-flash` 切换为
  `glm-4-flash-250414`。同一短提示三次实测首 token 为 195–331ms，
  完整生成 534–883ms；备用模型为 `glm-4-flash`。
- 页面打开、重新回到前台或开始输入时，会在后台预热 BGE-M3；连续无操作后仍按
  180 秒释放 RAG GPU 工作进程。冷预热实测约 1.9–3.6 秒，但不阻塞页面操作。
- 公网预热后的真实流式问答实测：检索 27ms、首文本 1033ms、首个完整语义句
  1490ms、文本生成完成 2279ms。
- 更换模型后标准知识题仍为 15/15；该结果是关键词自动预检基线，不替代专家事实准确率评审。

## 测试口径

基准脚本在文字请求开始计时前先建立 WebRTC 会话，然后发送流式
`POST /v1/chat`，并持续消费真实 WebRTC 音轨。测试同时查询服务端 TTS
工作队列和 LiveTalking 的说话状态，直到后台队列结束、数字人停止说话且音轨
连续静音 800ms，最后关闭测试会话。

因此本报告衡量的是页面数字人已经连接时的：

`发送文字 → LLM 流式生成 → 分句 TTS → Wav2Lip → WebRTC 音频最后一帧`

不包含首次打开页面、LiveTalking CPU→GPU 迁移或公网 TURN 建连时间。

脚本输出字段：

- `first_token_ms`：收到第一个 LLM 文本增量。
- `chat_done_ms`：LLM 文本生成结束。
- `first_audio_ms`：收到第一个非静音 WebRTC 音频帧，即用户体感首响。
- `speech_queue_done_ms`：所有分句完成 TTS 并送入 LiveTalking 队列。
- `last_audio_ms`：收到最后一个非静音 WebRTC 音频帧。
- `completion_observed_ms`：队列、说话状态和静音窗口共同确认播报结束。

复测命令：

```bash
cd /home/gmn/codes/cup
/home/gmn/.conda/envs/ccc/bin/python \
  services/api/scripts/benchmark_opening_latency.py \
  http://127.0.0.1:8001 \
  --question '灵山大佛有多高？' \
  --model-route cloud
```

## 当前低延迟配置

针对实听中出现的非语义停顿、偶发吞字和倍速问题，当前实现为：

1. 首个不少于 8 字的逗号完整短句优先播报；后续逗号短句需达到 24 字，
   120 字无标点才使用安全切分。
2. GLM-TTS 的 PCM 块边生成边送入同一个回答级有状态重采样流，不再等待整段合并成
   WAV，也不会在每个句子之间重置重采样、补零或重复淡入淡出。
3. 播报速度固定为自然原速 1.0，不再启用 1.12 倍速。

由 systemd 托管的新进程完成 RAG 预热后，连续测试 4 次，中位数如下：

| 指标 | 当前结果 |
|---|---:|
| LLM 首 token | 562ms |
| LLM 结束 | 898ms |
| 首个非静音音频 | 2160ms |
| TTS 队列完成 | 3021ms |
| 最后非静音音频 | 8799ms |
| 首 token → 首响 | 1587ms |
| 实际播报跨度 | 6803ms |
| 回答长度 | 25 字 |

四轮均完整收到非静音音频，首句和后续句分别形成连续 PCM 段；服务日志没有 TTS、
PCM 或数字人队列错误，水印音过滤仍保持启用。

## 基线与历史延迟优先结果

短事实问题各测试 3 次，下表取中位数：

| 指标 | 优化前 | 优化后 | 变化 |
|---|---:|---:|---:|
| LLM 首 token | 767ms | 1344ms | +577ms |
| LLM 结束 | 1001ms | 1889ms | +888ms |
| 首个非静音音频 | 3459ms | 2987ms | -472ms（-13.6%） |
| TTS 队列完成 | 5364ms | 5057ms | -307ms（-5.7%） |
| 最后非静音音频 | 12189ms | 9387ms | -2802ms（-23.0%） |
| 首 token → 首响 | 2692ms | 1643ms | -1049ms（-39.0%） |
| 实际播报跨度 | 9052ms | 6400ms | -2652ms（-29.3%） |
| 回答长度 | 41 字 | 33 字 | -8 字 |

本轮复测期间项目的云端模型由 `glm-4.7-flash` 切换为
`glm-4.5-flash`，外部接口波动也使优化后 LLM 首 token 反而变慢。即便如此，
端到端首响和完成时间仍然下降；用“首 token → 首响”隔离模型网络波动后，
TTS/数字人首响阶段缩短约 39%。

## 历史延迟优先措施（第 1、2 项已回退）

1. 曾将每个 GLM-TTS PCM 流式块立即封装 WAV 并上传 LiveTalking；该方案会在
   非语义边界重复重采样和淡入淡出，现已改为完整语义段上传。
2. 曾把 LiveTalking 口播速度设为 1.12；现已固定恢复为自然原速 1.0。
3. RAG 提示要求普通事实回答通常不超过 50 字、1–3 句，第一句尽量不超过
   18 字；路线、攻略和用户明确要求详细介绍时仍可展开。
4. `/v1/livetalking/is-speaking` 增加后台语音队列 `pending` 状态。前端只有在
   LLM 已结束、队列已清空且实际说话停止后才显示“播报完成”，避免把句间短暂
   静音误判为结束。
5. 基准脚本从只测首响升级为持续采集最后音频帧，并支持
   `--livetalking-url` 指定隔离测试实例。

`first_audio_ms` 对照。
