# 文字到语音端到端延迟测试与优化

测试日期：2026-07-19

## 测试口径

基准脚本在文字请求开始计时前先建立 WebRTC 会话，然后发送流式
`POST /v1/chat`，并持续消费真实 WebRTC 音轨。测试同时查询服务端 TTS
工作队列和 LiveTalking 的说话状态，直到后台队列结束、数字人停止说话且音轨
连续静音 800ms，最后关闭测试会话。

因此本报告衡量的是页面数字人已经连接时的：

`发送文字 → LLM 流式生成 → 分句 TTS → Wav2Lip → WebRTC 音频最后一帧`

不包含首次打开页面、LiveTalking CPU→GPU 迁移或公网 TURN 建连时间。

脚本输出字段：

- `first_token_ms`：收到第一个 LLM 文本增量。
- `chat_done_ms`：LLM 文本生成结束。
- `first_audio_ms`：收到第一个非静音 WebRTC 音频帧，即用户体感首响。
- `speech_queue_done_ms`：所有分句完成 TTS 并送入 LiveTalking 队列。
- `last_audio_ms`：收到最后一个非静音 WebRTC 音频帧。
- `completion_observed_ms`：队列、说话状态和静音窗口共同确认播报结束。

复测命令：

```bash
cd /home/gmn/codes/cup

  services/api/scripts/benchmark_opening_latency.py \
  http://127.0.0.1:8001 \

```

## 当前质量优先配置

针对实听中出现的非语义停顿、偶发吞字和倍速问题，当前实现已调整为：

1. 普通文本不再按 12 字硬切；优先等待句末标点，较长内容只在自然分句处切分，
   120 字无标点才使用安全切分。
2. GLM-TTS 的多个 PCM 流式块先合并为完整语义段，只向 LiveTalking 上传一份 WAV，
   避免每个提供方分块被单独重采样和淡入淡出。
3. 播报速度固定为自然原速 1.0，不再启用 1.12 倍速。

重启后热链路连续测试 3 次，中位数如下：

| 指标 | 当前结果 |
|---|---:|
| LLM 首 token | 535ms |
| LLM 结束 | 1009ms |
| 首个非静音音频 | 3618ms |
| TTS 队列完成 | 3656ms |
| 最后非静音音频 | 10773ms |
| 首 token → 首响 | 3121ms |
| 实际播报跨度 | 7214ms |
| 回答长度 | 33 字 |

三轮均完整收到非静音音频，服务日志没有 TTS 或数字人队列错误。由于改为等待完整
语义段再上传且恢复原速，首响和播报时长会比下面的历史激进分块模式略长；这是为
保证断句、尾字和整体可懂度所做的明确取舍。

## 基线与历史延迟优先结果

短事实问题各测试 3 次，下表取中位数：

| 指标 | 优化前 | 优化后 | 变化 |
|---|---:|---:|---:|
| LLM 首 token | 767ms | 1344ms | +577ms |
| LLM 结束 | 1001ms | 1889ms | +888ms |
| 首个非静音音频 | 3459ms | 2987ms | -472ms（-13.6%） |
| TTS 队列完成 | 5364ms | 5057ms | -307ms（-5.7%） |
| 最后非静音音频 | 12189ms | 9387ms | -2802ms（-23.0%） |
| 首 token → 首响 | 2692ms | 1643ms | -1049ms（-39.0%） |
| 实际播报跨度 | 9052ms | 6400ms | -2652ms（-29.3%） |
| 回答长度 | 41 字 | 33 字 | -8 字 |

本轮复测期间项目的云端模型由 `glm-4.7-flash` 切换为
`glm-4.5-flash`，外部接口波动也使优化后 LLM 首 token 反而变慢。即便如此，
端到端首响和完成时间仍然下降；用“首 token → 首响”隔离模型网络波动后，
TTS/数字人首响阶段缩短约 39%。

## 历史延迟优先措施（第 1、2 项已回退）

1. 曾将每个 GLM-TTS PCM 流式块立即封装 WAV 并上传 LiveTalking；该方案会在
   非语义边界重复重采样和淡入淡出，现已改为完整语义段上传。
2. 曾把 LiveTalking 口播速度设为 1.12；现已固定恢复为自然原速 1.0。
3. RAG 提示要求普通事实回答通常不超过 50 字、1–3 句，第一句尽量不超过
   18 字；路线、攻略和用户明确要求详细介绍时仍可展开。
4. `/v1/livetalking/is-speaking` 增加后台语音队列 `pending` 状态。前端只有在
   LLM 已结束、队列已清空且实际说话停止后才显示“播报完成”，避免把句间短暂
   静音误判为结束。
5. 基准脚本从只测首响升级为持续采集最后音频帧，并支持
   `--livetalking-url` 指定隔离测试实例。

## 长回答验证

问题“请详细介绍灵山梵宫的建筑特色和游览建议。”生成 154 字：

| 指标 | 结果 |
|---|---:|
| LLM 首 token | 2709ms |
| LLM 结束 | 5237ms |
| 首个非静音音频 | 4651ms |
| TTS 队列完成 | 23084ms |
| 最后非静音音频 | 42430ms |
| 完成状态确认 | 43237ms |

长回答在 LLM 尚未完成时已开始播报，且测试会话在最后音频帧和完成确认后才关闭，
验证了多句队列不会被句间静音提前结束。
