# 景区导览服务 AI 数字人总体设计文档

## 1. 建设目标

项目面向灵山胜境游客和景区管理者，提供 7×24 小时可交互的 AI 数字人导游。
游客能够通过语音、文字和图片获得可靠讲解与个性化路线；管理者能够维护知识、
配置数字人并查看游客关注点、满意度和情感趋势。

## 2. 总体架构

```text
游客浏览器（HTTP/HTTPS）
  ├─ 文字 / 麦克风 / 图片 / 兴趣 / 位置
  └─ WebRTC 数字人音视频
             │
             ▼
FastAPI 导览网关（8001 / 8443）
  ├─ ASR：GLM-ASR
  ├─ RAG 服务（8020，仅内部）：BGE-M3 + FAISS + 有界会话历史
  ├─ LLM：GLM-4.7-Flash API / 本地 Qwen2-7B（流式增量输出）
  ├─ Vision：GLM-4V 识景后再次进行知识库校准
  ├─ TTS：GLM-TTS PCM 分块流式合成
  ├─ SQLite：交互日志、反馈、情感任务、数字人配置
  ├─ 洞察引擎：实时日志聚合 + XLSX 历史游客行为分析
  └─ 管理后台：知识库、数字人、运营分析、多模态感受度报告
             │
             ├─ LiveTalking（8010，仅内部）→ Wav2Lip FP16 → WebRTC
             └─ HumanOmni 七分类模型 / 独立情感推理服务（异步）
```

## 3. 功能设计

### 3.1 游客交互端

1. 文字与语音输入：浏览器录音转 16kHz WAV，调用 GLM-ASR，再进入统一问答链路。
2. 景区知识问答：回答必须基于检索资料，资料不足时明确拒绝编造。
3. 数字人回答：首个完整短句生成后立即进入 TTS 与口型流水线，不等待全文完成。
4. 个性化路线：按历史、自然、亲子兴趣返回不同路线、时长和讲解重点。
5. 多模态识景：上传或拍摄图片，由视觉模型识别，再用本地知识库校准讲解。
6. 弱定位：支持 GPS、二维码、Wi-Fi 节点和手动选择的降级定位接口。
7. 满意度反馈：按景区/子景点采集 1–5 分及文字意见，进入满意度趋势和服务建议。

### 3.2 管理后台

1. 数据概览：今日/近七日服务游客、累计问答、响应时间、满意度、热门咨询。
2. 知识库管理：上传或同名更新 DOCX/TXT/Markdown，删除管理员文档，自动重建向量索引。
3. 数字人管理：维护名称、形象、声音、服装风格和默认表情，新会话生效。
4. 感受度报告：只读展示游客对话产生的七类语音情绪、文本倾向、景点满意度、关注方面和有证据的运营建议。
5. 历史游客洞察：解析赛题 XLSX，展示满意度分布和月度趋势，并按景点类型、年龄、消费区间分析差异。

## 4. 核心技术

### 4.1 可靠 RAG

- 官方 DOCX、结构化 DOCX、XLSX 和管理员资料按标题、段落切分，长片段保留句级重叠。
- BGE-M3 生成归一化向量，FAISS `IndexFlatIP` 执行余弦相似度检索；命中景点名或别名时先按景区元数据缩小候选集。
- RAG 作为单独的本机服务持有模型和索引，多个游客/管理 API 监听器无需重复加载 BGE-M3。
- 系统提示词约束数字、日期、重量等事实必须来自检索上下文。
- 流式响应先返回稳定引用 ID，再输出文本增量；回答可直接进入按句 TTS 与 LiveTalking。
- 不提供会话 ID 时自动创建新 ID；后续携带相同 ID 可理解代词追问。历史受 TTL、会话数和轮数限制，取消或失败的半截回答不入库。

### 4.2 低延迟语音数字人

系统采用 ASR → LLM 语义句 → 完整语义段 TTS → Wav2Lip 的流水线。LLM 输出优先按
句号、问号等语义边界切句，不按固定短字数截断；TTS 流式 PCM 在每个语义段内完整
合并后再送入数字人，并以 1.0 原速播放。Wav2Lip 使用 FP16 和 batch 8，在保证口型
表现的同时降低显存占用。

### 4.3 游客洞察

该模块分为实时运营、历史游客行为、多模态感受度三条数据链。实时运营从聊天和反馈
记录聚合服务人次、响应时延、热门问题与近七日趋势；历史分析将赛题 XLSX 明细按
源文件指纹幂等导入 SQLite，并以流式方式计算满意度分布、月度变化、客群/消费/景点
类型分层及相关性，聚合结果按源文件版本缓存。

多模态感受度分析以游客当前语音、ASR 文本和最近对话上下文为输入；管理员后台不
提供样本上传入口，也不分析数字人的合成语音。HumanOmni 微调模型输出
`angry/disgust/fear/happy/neutral/sad/surprise` 七类情绪及
置信度；文本模块同时提取讲解内容、回答准确性、响应速度、路线推荐、票价、餐饮、
交通等方面。融合层根据实际可用模态动态分配权重，将七类情绪证据、文本倾向与显式
显式景点评分作为独立信号展示。七类情绪与满意度是不同任务，系统分别保存和
展示，不把 XLSX 的满意度数值伪装成音视频情绪训练标签。

情感推理与游客问答链路隔离并采用后台任务，模型不可用、输出异常或超时时自动降级
为 ASR 文本结果，同时在后台显示实际分析模式和错误原因。默认分析后删除原始音频，
仅持久化结构化结果；游客端必须主动授权原声情绪分析。

## 5. 数据设计

- `chat_logs`：会话、角色、内容、检索/情感/延迟元数据、时间。
- `feedback`：会话、景区、子景点、1–5 分、意见、情感类别、来源、时间。
- `emotion_events`：多模态任务状态、七类情绪概率、融合满意倾向、方面、模型与错误信息。
- `attractions`：来自赛题 DOCX 的景区/子景点稳定 ID 与名称。
- `tourism_visits`：赛题 XLSX 的 17 个原始业务字段、源文件和行号，共 140447 行。
- `dataset_imports`：历史数据源指纹、导入行数和导入时间。
- `avatar_settings`：数字人名称、模型、声音、服装、表情、更新时间。
- `tourism_analytics_cache.json`：XLSX 源文件指纹和聚合后的历史游客洞察缓存。
- `llm/index/faiss.index`：BGE-M3 向量索引。
- `llm/index/metadata.json`：与向量逐条对应的知识片段、来源、景区和小节元数据。
- RAG 会话：进程内 TTL/LRU 有界存储；服务重启即清空，不保存游客长期画像。

## 6. 非功能性设计

- 准确性：15 道标准事实题实测 100%，超过赛题要求的 90%。
- 延迟：语音结束至数字人首个非静音音频帧实测 3373ms，低于 5 秒目标。
- 稳定性：LiveTalking 不可用时游客端自动回退 Live2D，并每 10 秒重连。
- 隔离性：情感推理异步执行，不占用实时问答路径；模型故障时保留可解释的降级结果。
- 隐私性：多模态采集要求明确授权，限制文件类型和大小，默认不保留原始媒体。
- 可维护性：API、RAG、数字人和静态前端解耦；所有模型及端口可通过环境变量调整。
- 安全性：模型密钥仅保存在服务端；RAG/LiveTalking 只监听内部端口，索引重建和
  知识文档修改只允许登录后的独立管理端口访问；XMOV 浏览器密钥默认不下发。

## 7. 创新点

1. “首句即说”的跨模型流水线，把等待全文的串行链路改造成按句并行链路。
2. 视觉识别结果再次经过本地知识库校准，减少通用视觉模型的景点幻觉。
3. 网络定位失效时提供二维码、Wi-Fi 和手动选择的多级降级策略。
4. 将七类音视频情绪、方面级文本分析和显式满意度动态融合，保留来源、置信度和降级状态。
5. 实时交互数据与赛题历史 XLSX 双层分析，既支持当日运营，也支持客群和消费决策。

## 8. 当前边界与演进

- 当前提供一个 Wav2Lip 真人头像；可通过管理端和头像生成接口扩充形象。
- GPS/Wi-Fi 为可替换的演示适配层，生产部署需接入真实坐标和锐捷网络位置数据。
- 情感模型路径和启动脚本因服务器部署方式而异；可通过 HTTP 推理服务或本地脚本接入。
  模型缺失时系统会明确标记降级，不能据此宣称完成了真实音视频模型评测。
- XLSX 的 `satisfaction` 只作为历史满意度统计与融合校准信号，不能直接用于训练七类情绪分类器；
  若需针对景区微调，应另行采集取得授权且含音频/视频、文本和情绪标注的数据。
